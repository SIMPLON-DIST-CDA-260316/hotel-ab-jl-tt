  ---
  Dockerfile — Issues

  1. Volume /app/data inutile (lignes 28-29)
  C'est un vestige de SQLite. Avec PostgreSQL, les données sont dans le
  container db, pas dans l'app.

  # À supprimer :

  VOLUME /app/data

  2. HEALTHCHECK fragile
  bun -e "fetch(...)" peut échouer si le runtime Bun n'est pas dans le PATH ou
  si fetch n'est pas disponible dans bun:1-slim. Préférer curl ou wget :

  # Alternative plus robuste :
  HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/ || exit 1

  Mais bun:1-slim ne contient probablement ni curl ni wget. Deux options :
  - Installer wget (RUN apt-get update && apt-get install -y wget && rm -rf 
  /var/lib/apt/lists/*)
  - Garder la version Bun si elle fonctionne (à tester)

  3. Pas de .env dans le build context
  Le .dockerignore exclut correctement .env, c'est bien. Mais attention : COPY .
   . dans le stage builder copie .env.example — pas bloquant mais à noter.

  ---
  docker-compose.yml — Issues

  4. Credentials PostgreSQL en dur (lignes 5-7)
  Les credentials sont en clair dans le compose. Pour un exercice c'est
  acceptable, mais en prod il faudrait utiliser des variables d'environnement ou
   Docker secrets.

  5. DATABASE_URL dans l'app doit pointer vers db, pas localhost
  Le .env.example montre localhost:5432, ce qui fonctionne en dev local mais pas
   dans Docker Compose. Quand l'app tourne dans un container, elle doit
  atteindre le service db par son nom de service :

  C'est le bug le plus critique — l'app ne pourra pas se connecter à la DB en
  mode Docker.

  6. Port 5432 exposé sur l'hôte (ligne 9)
  Le port Postgres est exposé publiquement. Pour le dev c'est pratique (accès
  avec un client SQL local), mais en prod il faudrait le retirer ou utiliser un
  réseau interne.

  7. Pas de réseau custom
  Ajouter un réseau dédié isolerait les services :
  networks:
    backend:
      driver: bridge

  ---
  Schema Migration (SQLite → PostgreSQL)

  8. timestamp vs timestamp_ms
  Le passage de integer("...", { mode: "timestamp_ms" }) à timestamp("...") est
  correct. Drizzle gère bien le mapping. Mais vérifier que better-auth attend
  bien des timestamp PostgreSQL natifs (et pas des integers).

  9. drizzle-orm/node-postgres au lieu de drizzle-orm/bun-sql
  Tu utilises Bun comme runtime (Dockerfile) mais node-postgres (package pg)
  comme driver. Ça fonctionne car Bun est compatible Node, mais tu pourrais
  aussi utiliser drizzle-orm/bun-sql avec le driver natif Bun pour PostgreSQL

● Voici l'architecture Oryjin que j'ai inspectée :

  Structure résumée

  src/
  ├── components/      # UI partagé (layout, form, ui/shadcn)
  ├── hooks/           # Hooks partagés (use-mobile, use-locale...)
  ├── lib/             # Libs partagées (api client, query client, config)
  ├── types/           # Types partagés (api.types, domain.types)
  ├── stores/          # État global UI uniquement (Zustand)
  ├── features/        # Modules métier auto-contenus
  │   ├── auth/        →  components/ + hooks/ + queries
  │   ├── chat/        →  components/ + hooks/ + lib/ + types
  │   ├── projects/    →  components/ + hooks/ + lib/
  │   └── activity/    →  components/ + hooks/
  └── routes/          # Couche app (TanStack Router)

  Règles d'import

  shared (components, hooks, lib, types)
      ↓
  features (importent du shared, JAMAIS entre features)
      ↓
  routes (composent features + shared)

  - Pas de barrel files (pas d'index.ts) — pour la perf Vite
  - Layouts dumb — reçoivent les données via props depuis les routes
  - Query hooks co-localisés dans chaque feature avec queryOptions factories

  ---
  Est-ce compatible avec ton projet Next.js hotel-exo ?

  Oui, c'est compatible, mais avec des adaptations car Next.js App Router change
   la donne :

  ┌─────────────┬────────────────────────────┬──────────────────────────────┐
  │   Aspect    │  Oryjin (Vite + TanStack   │   Hotel-exo (Next.js App     │
  │             │          Router)           │           Router)            │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Routing     │ src/routes/ fichiers       │ app/ fichiers Next.js        │
  │             │ TanStack                   │                              │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Data        │ TanStack Query (client)    │ Server Components + Server   │
  │ fetching    │                            │ Actions                      │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ État        │ TanStack Query             │ Souvent inutile (RSC fait le │
  │ serveur     │                            │  job)                        │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Couche      │ routes/ compose features   │ app/ compose features        │
  │ "app"       │                            │                              │
  └─────────────┴────────────────────────────┴──────────────────────────────┘

  Ce qui s'adapte bien

  - features/ avec components/, hooks/, lib/ → identique
  - components/ partagés (UI, layout) → identique
  - lib/ (utils, config) → identique
  - types/ partagés → identique
  - Règle d'import shared → feature → app → identique

  Ce qui change

  - Pas besoin de TanStack Query dans la plupart des cas — les Server Components
   fetch directement
  - Server Actions remplacent les mutations API côté client
  - stores/ quasi inutile — moins d'état client avec RSC
  - La couche "routes" c'est app/ directement (convention Next.js)

  Mon avis

  Pour un projet d'apprentissage CDA, oui ça vaut le coup d'adopter cette
  structure. Ça te donne une architecture propre et scalable. La version Next.js
   ressemblerait à :

  src/
  ├── components/      # UI partagé
  ├── lib/             # Utils, config, db
  ├── types/           # Types partagés
  ├── features/
  │   ├── auth/        →  components/ + actions/ + queries/
  │   ├── rooms/       →  components/ + actions/ + queries/
  │   └── bookings/    →  components/ + actions/ + queries/
  app/
  ├── page.tsx         # Compose depuis features/
  ├── rooms/
  │   └── page.tsx     # Importe depuis features/rooms
  └── layout.tsx

  Tu veux que je mette ça en place sur hotel-exo ?
